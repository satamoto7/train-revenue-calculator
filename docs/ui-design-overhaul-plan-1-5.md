# UI/デザイン全面見直し計画（1〜5）

本ドキュメントは、既存アプリの UI/デザイン見直しに向けた優先課題 **1〜5** を実装可能なレベルまで整理した計画書です。  
対象は主に `src/App.js`（現行の単一ファイルUI）と `tailwind.config.js` です。

---

## 1. デザインシステム未整備の解消

### 背景
- ボタン・入力・カード・モーダルの見た目定義が個別実装になっており、画面間でスタイルの一貫性が崩れやすい。
- Tailwind クラスの重複・バリエーション乱立により、改修時の影響範囲が読みづらい。

### 目標
- プロジェクト共通の「見た目の基準」を先に定義し、UI 実装の判断を統一する。

### 実施内容
1. 現行 UI パーツを棚卸し（Button/Input/Card/Modal/Section Header）
2. バリアント定義（Primary/Secondary/Danger/Ghost/Inline）
3. `tailwind.config.js` にデザイントークン追加（色・余白・角丸・シャドウ・フォント）
4. `src/components/ui/` に共通コンポーネントを作成し、`src/App.js` から段階置換

### 完了条件
- 新規 UI は必ず共通コンポーネント経由で実装される。
- 同種コンポーネントの見た目差分がバリアントに集約される。

---

## 2. 情報設計（IA）の再設計

### 背景
- 管理・入力・結果確認が同じ画面上で混在し、ユーザーの認知負荷が高い。
- セクションの開閉状態が文脈依存で、初見ユーザーが「どこから触るべきか」を判断しにくい。

### 目標
- 画面ごとの主目的を明確にし、UI 構造を「迷わない順序」に並べ替える。

### 実施内容
1. 既存セクションを「設定」「入力」「確認」に分類
2. 利用頻度・重要度順にセクションを再配置
3. 破壊操作（削除/リセット）を下部へ分離
4. デフォルト開閉状態を設計（初回は最重要だけ展開）
5. `src/views/management/` / `src/views/company-detail/` などへ責務分割

### 完了条件
- ユーザーが上から順に操作するだけで主要タスクを完了できる。
- セクション構成が画面目的と一致する。

---

## 3. ナビゲーションと初回導線の強化

### 背景
- タブ遷移はあるが、空データ時の「次に何をすべきか」が弱い。
- 初回利用時のセットアップ手順が UI 上で明示されていない。

### 目標
- 現在地と次アクションが常にわかるナビゲーションを提供する。

### 実施内容
1. 空データ時にセットアップガイドを表示（例: プレイヤー追加→企業追加→収益入力）
2. `SummaryView` の空状態に CTA ボタンを配置
3. タブに進捗表示（未完了/完了）を追加
4. 初回・既存データ時で初期表示ロジックを明確化
5. `src/App.test.js` に導線テストを追加

### 完了条件
- 初回ユーザーがヘルプなしで初期設定を終えられる。
- 各タブで「次にやること」が可視化される。

---

## 4. タイポグラフィと余白スケールの標準化

### 背景
- `text-xs` の多用などで、主要情報と補助情報の視覚階層が弱い。
- コンポーネントごとの文字サイズ・余白ルールが統一されていない。

### 目標
- 可読性を高めるため、文字・行間・余白の基準を統一する。

### 実施内容
1. 文言タイプ別（見出し/本文/補助/数値）にサイズを定義
2. `tailwind.config.js` の `fontSize`・`spacing` をプロジェクト向けに整理
3. 主要入力 UI（収益編集、割合入力、モーダル、企業ピル）を優先調整
4. モバイル幅（320px）での視認性確認項目を定義

### 完了条件
- どの画面でも見出し・本文・補助情報の階層が一貫して判別できる。
- 文字が小さすぎるために操作を誤るケースが解消される。

---

## 5. 色設計（ステータスカラー）の再定義

### 背景
- 成功・警告・危険・通常操作の色使いが文脈依存で、意味の一貫性が弱い。
- 色だけで意味を伝える箇所があり、判別性・アクセシビリティ上の課題がある。

### 目標
- 色の意味を固定し、状態認識を速く・正確にする。

### 実施内容
1. 現行色の利用箇所を棚卸し（green/red/blue/indigo/slate/purple など）
2. 役割ベースでトークン化（Info/Success/Warning/Danger/Neutral）
3. 破壊操作は Danger 系に統一、通常編集は Primary/Neutral に寄せる
4. 色依存の操作にはテキストラベルを併記
5. コントラスト観点（背景色×文字色）のチェック項目を追加

### 完了条件
- 同じ意味の操作・状態は、どの画面でも同じ色体系で表現される。
- 色覚差があってもラベル併記で状態を理解できる。

---

## 実装順（推奨）
1. **1: デザインシステム整備**
2. **4: タイポグラフィ/余白標準化**
3. **5: 色設計再定義**
4. **2: 情報設計再構成**
5. **3: ナビゲーション/初回導線強化**

> 理由: 先に見た目の基盤（1/4/5）を揃えると、IA と導線改修（2/3）で手戻りが減るため。
